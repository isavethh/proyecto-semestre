Este documento define el diseño del flujo completo de REGISTRO DE EMBAJADOR, pensado para ser replicado exactamente en Figma, pantalla por pantalla, con campos, estados, validaciones y reglas de negocio claras. El objetivo es que el programador y el diseñador no interpreten: solo implementen. El flujo contempla dos escenarios: 1) Embajador que fue invitado por otro embajador. 2) Embajador que se registra solo y pasa directamente a la red de la empresa. PANTALLA 0 – ENTRY POINT / ATRIBUCIÓN DE INVITACIÓN Tipo de pantalla: Pantalla invisible / lógica previa (no siempre visible al usuario). Comportamiento esperado: - El sistema detecta si el usuario llega desde: a) Link o QR con parámetro ref b) Sin parámetro ref Elementos lógicos (no UI): - ref_id (string | opcional) - source (link, qr, manual, organic) Estados: - Si ref_id existe y es válido → se guarda en sesión - Si no existe → se marca como registro directo a la empresa Regla de negocio (para el programador): Este paso define la atribución inicial. Si existe un ref_id válido, el embajador queda asociado al invitador como Nivel 1. Si no existe, el embajador queda asociado directamente a la red de la empresa, sin invitador humano. Esta decisión es inmutable una vez finalizado el registro. La atribución nunca debe depender solo de inputs posteriores. El sistema debe priorizar el contexto de entrada para evitar fraude, cambios manuales o conflictos de atribución. PANTALLA 1 – REGISTRO BÁSICO DEL EMBAJADOR Tipo de pantalla: Formulario principal (Figma frame 1). Componentes UI: [ Logo de la empresa ] Campo: Nombre completo - Input text - Obligatorio Campo: Correo electrónico - Input email - Obligatorio - Validación formato Campo: Teléfono - Input numérico - Obligatorio Campo: Contraseña - Input password - Reglas visibles: mínimo 8 caracteres Campo: Confirmar contraseña Checkbox: [ ] Acepto términos y condiciones Botón primario: [ Crear cuenta de embajador ] Regla de negocio: En esta pantalla solo se crea la identidad básica del embajador. No se calculan comisiones ni niveles aún. El registro no se considera completo hasta validar correo y aceptar términos. El sistema debe validar unicidad de correo y teléfono. Si el usuario ya existe como cliente o conductor, se debe enlazar la identidad, no crear un duplicado. PANTALLA 2 – IDENTIFICACIÓN DE INVITACIÓN (VISIBLE SOLO SI NO HAY ref_id) Tipo de pantalla: Formulario condicional. Texto superior: “¿Alguien te recomendó la plataforma?” Campo opcional: - Código de invitación - Input text - Placeholder: “Ej: AF-12345” Texto ayuda: “Si alguien te recomendó, puedes ingresar su código para que reciba el reconocimiento correspondiente.” Botón: [ Continuar ] Estados: - Si el código es válido → mostrar nombre/alias del invitador (solo confirmación) - Si no se ingresa nada → continuar como registro directo Regla de negocio: Este campo es opcional y solo actúa como respaldo. Si el usuario ya llegó con ref_id, esta pantalla no debe mostrarse. Una vez confirmado este paso, la atribución queda bloqueada. El usuario no puede cambiar quién lo invitó después del registro. PANTALLA 3 – TIPO DE PARTICIPACIÓN Tipo de pantalla: Selección simple (radio cards). Opciones: ( ) Quiero recomendar restaurantes ( ) Quiero recomendar conductores ( ) Quiero recomendar ambos Texto aclaratorio: “Puedes cambiar estas preferencias más adelante.” Botón: [ Continuar ] Regla de negocio: Esta pantalla no define permisos finales, solo preferencias iniciales. Sirve para personalizar el onboarding y las primeras sugerencias de IA. No debe limitar funcionalidades futuras ni afectar el cálculo de comisiones. PANTALLA 4 – CONFIRMACIÓN DE ATRIBUCIÓN Tipo de pantalla: Confirmación informativa. Bloque informativo: Si tiene invitador: “Estás siendo registrado como embajador recomendado por: Carlos M.” Si es registro directo: “Te estás registrando como embajador independiente dentro de la red de la empresa.” Texto legal pequeño: “La atribución de recomendación no puede modificarse una vez completado el registro.” Botón: [ Confirmar y continuar ] Regla de negocio: Esta pantalla existe para eliminar conflictos posteriores. El usuario confirma explícitamente la atribución antes de finalizar el alta. El sistema debe guardar un log de aceptación para auditoría futura. PANTALLA 5 – DATOS DE PAGO Tipo de pantalla: Formulario financiero. Campos: - Tipo de pago (transferencia / billetera / otro) - Banco - Número de cuenta - Nombre del titular Texto aclaratorio: “Los pagos se realizan semanalmente. Puedes modificar estos datos en cualquier momento.” Botón: [ Guardar y continuar ] Regla de negocio: Los datos de pago no son obligatorios para completar el registro, pero sí para recibir pagos. El sistema debe permitir avanzar aunque estén incompletos, pero bloquear liquidaciones si faltan. PANTALLA 6 – REGISTRO COMPLETADO Tipo de pantalla: Pantalla de éxito. Mensaje principal: “Tu cuenta de embajador ha sido creada con éxito.” Subtexto: “Desde tu panel podrás ver el impacto de tus recomendaciones y tus ingresos en tiempo real.” Botón: [ Ir a mi backoffice ] Regla de negocio: En este punto el embajador ya existe en el sistema y puede generar impacto. El acceso al backoffice debe estar habilitado incluso si aún no tiene actividad. REGLAS DE NEGOCIO – PARA EL PROGRAMADOR Regla de negocio 1 (verificación y permisos): La verificación de correo y teléfono es obligatoria para habilitar pagos, comisiones y liquidaciones. Mientras uno o ambos estados estén en “Pendiente”, el embajador puede acceder solo a vistas limitadas del backoffice (lectura, sin acciones financieras). El sistema debe validar estos estados en cada request sensible (pagos, reportes financieros, invitaciones). La verificación debe ser idempotente y auditable. Cada evento (envío de correo, validación de link, envío de OTP, validación de OTP) debe registrarse con timestamp, IP y dispositivo para soporte y prevención de fraude. Regla de negocio 2 (experiencia y seguridad): El sistema debe permitir reenvíos controlados (cooldown) tanto para correo como SMS, evitando spam y abusos. Los códigos OTP deben expirar (ej. 5 minutos) y tener un número máximo de intentos. Una vez que ambos canales estén verificados, el estado de la cuenta pasa a “Verificada” y se habilita automáticamente el siguiente paso del onboarding (Completar perfil del embajador). Esta transición debe ser automática y no manual. REGLAS DE NEGOCIO – PARA EL PROGRAMADOR Regla de negocio 1 (habilitación de pagos): El sistema no debe ejecutar liquidaciones ni pagos semanales a ningún embajador que no tenga un método de pago configurado y validado. Sin embargo, el embajador sí puede ver sus ingresos acumulados en tiempo real. Las ganancias deben quedar en estado “Pendiente de pago” hasta que el método esté completo. Una vez configurado, las liquidaciones futuras se procesan automáticamente en el siguiente ciclo semanal. Regla de negocio 2 (edición y auditoría): El embajador puede editar su método de pago en cualquier momento, pero cada cambio debe quedar auditado (fecha, valor anterior, valor nuevo). Para evitar fraudes, la empresa puede definir ventanas de bloqueo (por ejemplo, no permitir cambios 24h antes del cierre semanal). El sistema debe validar que el titular del método coincida con el embajador registrado, salvo excepciones explícitas definidas por la empresa. CONCEPTO CENTRAL DEL APP EMBAJADOR El APP Embajador NO es una app de negocio. El APP Embajador NO es obligatorio. El APP Embajador NO gestiona operaciones. El APP Embajador es un microservicio financiero-comercial cuya única función es: - Visualizar - Calcular - Auditar – Cobrar ingresos generados por recomendaciones dentro del ecosistema . Esto desacopla completamente la experiencia operativa del negocio (restaurante, transporte, delivery, etc.) de la experiencia de ingresos por red. DESCARGA Y ACCESO El APP Embajador: - Se descarga de forma independiente - No viene preinstalado ni forzado desde ningún negocio - No es requerido para usar ningún servicio de la empresa Un usuario (persona o empresa) puede: - Usar restaurantes, transporte o delivery - Tener negocios activos - Operar normalmente SIN tener jamás el APP Embajador instalado. Esto elimina cualquier percepción de obligación de hacer redes de mercadeo. QUIÉN USA EL APP EMBAJADOR Pueden usar el APP Embajador: - Personas naturales (clientes) - Restaurantes - Empresas de cualquier vertical Condición: - Haber sido recomendador de al menos un registro - O estar debajo de alguien que recomendó El APP Embajador no distingue jerarquía social. PRINCIPIO MAESTRO DE DECISIÓN La Inteligencia Artificial no debe ser tratada como una feature decorativa ni como un argumento de marketing. La regla fundamental es la siguiente: toda IA que no genere ingresos directos, ahorro de tiempo, reducción de fricción o claridad operativa para el usuario, restaurante, conductor o afiliado, no debe implementarse en la versión inicial. La IA no es el producto; es un acelerador de valor. Si el usuario no percibe utilidad clara en menos de cinco segundos de uso, esa IA se convierte en ruido. MATRIZ DE PRIORIDAD DE IA Las funcionalidades se clasifican en tres niveles: Prioridad Alta (obligatorias en la v1), Prioridad Media (v2 o versiones premium) y Prioridad Baja (no implementar). IA PARA USUARIOS (CONSUMIDORES) FUNCIONALIDADES CON PRIORIDAD ALTA (v1) Recomendaciones contextuales inteligentes Esta IA debe sugerir qué pedir en función de múltiples variables: hora del día, clima, ubicación del usuario, historial de pedidos y presupuesto aproximado. El objetivo es reducir la indecisión del usuario y aumentar la tasa de conversión. Ejemplo práctico: en un día caluroso, a las 13:30, la app recomienda jugos naturales y menús económicos cercanos al lugar del usuario. Esta IA se justifica porque genera uso inmediato, sensación de personalización real y aumento directo de pedidos. Ranking dinámico local La app debe mostrar rankings que cambien según la zona y el momento. Ejemplos: “Más pedidos ahora en Equipetrol”, “Top precio-calidad en tu barrio”, “Lo más pedido hoy cerca de ti”. Este enfoque es clave porque Santa Cruz es hiperlocal. Los usuarios confían más en lo que funciona en su zona que en rankings genéricos. Esta IA genera confianza y diferenciación frente a apps internacionales . Optimización de gasto La IA debe ayudar al usuario a gastar mejor, sugiriendo alternativas más económicas, combos convenientes o restaurantes con menor costo de delivery. En Bolivia, la sensibilidad al precio es alta, por lo que esta funcionalidad aporta valor inmediato y fidelización. FUNCIONALIDADES DE PRIORIDAD BAJA (NO v1) No deben implementarse en la primera versión: chatbots conversacionales tipo “habla con la app”, IA emocional, avatares virtuales o un chat tipo ChatGPT dentro de la app. Estas funciones no generan conversión ni uso sostenido en etapas tempranas. IA PARA RESTAURANTES (EJE CENTRAL DEL MODELO) FUNCIONALIDADES CON PRIORIDAD ALTA (v1) Predicción simple de demanda La IA debe analizar datos históricos para predecir picos de pedidos por día y hora. El sistema debe sugerir cuándo reforzar cocina, stock o delivery. No se busca precisión académica, sino utilidad operativa. Esta funcionalidad reduce pérdidas, mejora tiempos de entrega y posiciona a la plataforma como socio estratégico del restaurante. Recomendaciones de platos rentables La IA debe identificar qué platos se venden más y cuáles generan mayor margen. A partir de esto, debe sugerir qué platos destacar o cuáles ajustar en precio. Ejemplo: detectar un plato muy vendido pero con bajo margen y recomendar un ajuste de precio. Esta funcionalidad impacta directamente en los ingresos del restaurante, lo que justifica su permanencia en la plataforma. Sugerencias automáticas de promociones La IA debe detectar momentos de baja demanda y sugerir promociones específicas, como descuentos, combos o 2x1. También debe recomendar acciones en horarios de alta demanda. La mayoría de restaurantes no cuenta con conocimientos de marketing. Esta IA convierte a la plataforma en un asesor activo, no solo un canal de pedidos. FUNCIONALIDADES DE PRIORIDAD MEDIA O BAJA No deben implementarse en v1 análisis financieros avanzados tipo ERP, diseño automatizado de menús gourmet ni predicciones anuales complejas. Estas funciones pueden ofrecerse en planes premium futuros . IA PARA CONDUCTORES FUNCIONALIDADES CON PRIORIDAD ALTA (v1) Zonas calientes en tiempo real La IA debe mostrar mapas de calor simples indicando zonas con mayor demanda de viajes y pedidos, así como áreas con menor tiempo de espera. Esto incrementa los ingresos del conductor y reduce frustración, impactando directamente en la retención. Combinación inteligente de viajes y pedidos La IA debe sugerir al conductor secuencias eficientes, como entregar un pedido y luego tomar un viaje cercano. Esto reduce tiempos muertos y diferencia claramente a la plataforma frente a Uber o inDrive. FUNCIONALIDADES DE PRIORIDAD BAJA No deben implementarse IA de conducción, análisis psicológico del conductor ni sistemas de gamificación complejos, ya que no aportan valor inmediato. IA PARA AFILIADOS O EMBAJADORES FUNCIONALIDADES CON PRIORIDAD MEDIA (v1 SIMPLE) Recomendación de a quién invitar La IA debe sugerir al afiliado qué restaurantes o zonas aún no están cubiertas, ayudándolo a ser más efectivo. Esta funcionalidad profesionaliza el modelo de recomendación sin caer en esquemas agresivos. FUNCIONALIDADES A EVITAR No deben implementarse árboles multinivel visuales, rankings públicos de afiliados ni competencias agresivas, ya que refuerzan el estigma del marketing multinivel. IA QUE NO DEBE IMPLEMENTARSE Debe evitarse cualquier IA que solo sirva para marketing, que requiera demasiada explicación, que no tenga datos suficientes o que no se sienta en la experiencia diaria del usuario. ARQUITECTURA DE IA RECOMENDADA La arquitectura inicial debe basarse en reglas inteligentes combinadas con modelos de Machine Learning ligeros. No se recomienda usar modelos pesados en la etapa inicial. El enfoque correcto es comenzar con heurísticas claras y datos reales, y evolucionar progresivamente hacia modelos más complejos. PRINCIPIO ESTRATÉGICO BASE El backoffice del afiliado no debe comunicar jerarquía, poder ni competencia entre personas. Debe comunicar impacto, utilidad y crecimiento personal. El afiliado no debe sentirse parte de una “red”, sino de un programa profesional de generación de valor económico. El enfoque correcto es: resultados propios, impacto generado y oportunidades detectadas. REEMPLAZAR EL ÁRBOL MULTINIVEL POR UN MODELO DE IMPACTO PERSONAL Lo que NO se debe mostrar: - Árboles jerárquicos - Niveles visuales (oro, diamante, etc.) - Cantidad de personas debajo del afiliado Qué mostrar en su lugar: Panel de Impacto Personal Este panel responde a una sola pregunta clave: “¿Qué impacto económico estoy generando con mis recomendaciones?” Elementos visibles: - Restaurantes activos recomendados por el afiliado - Conductores activos recomendados - Usuarios activos recomendados - Ventas generadas por su red indirecta (sin mostrar personas) Ejemplo de texto: “Tus recomendaciones generaron Bs 12.450 en ventas este mes.” Esto desplaza el foco de personas hacia resultados reales. REEMPLAZAR RANKINGS PÚBLICOS POR PROGRESO PRIVADO Lo que NO se debe mostrar: - Top afiliados - Comparaciones entre personas - Posiciones públicas Qué mostrar en su lugar: Indicadores de Progreso Personal El afiliado solo ve su propio desempeño comparado contra: - Su mes anterior - Su promedio histórico Ejemplos: - “Este mes generaste 18% más impacto que el mes pasado.” - “Estás aprovechando el 65% de tu potencial según tu zona.” Esto mantiene motivación sin generar competencia tóxica. Ejemplos: - “Incorpora 3 restaurantes en una zona sin cobertura” - “Activa 5 conductores nuevos en horarios nocturnos” - “Ayuda a un restaurante a lanzar su primera promo” La recompensa no es protagonismo, sino: - Bonos claros - Beneficios operativos - Acceso a herramientas premium VISUALIZACIÓN CORRECTA DE INGRESOS El backoffice debe mostrar ingresos de forma clara, simple y auditada. ESTRUCTURA RECOMENDADA DEL BACKOFFICE DEL AFILIADO Secciones sugeridas: Resumen de Impacto Ingresos y Actividad Oportunidades Detectadas (IA) Misiones Activas Historial y Reportes Educación y Buenas Prácticas Cada sección debe reforzar profesionalismo, transparencia y utilidad. CONCLUSIÓN ESTRATÉGICA Un backoffice bien diseñado no necesita ocultar el modelo, necesita reinterpretarlo. Al eliminar árboles, rankings y competencias agresivas, y reemplazarlos por impacto, progreso personal y oportunidades reales, el programa deja de parecer marketing multinivel y pasa a percibirse como un sistema moderno de partners económicos. Este enfoque es especialmente clave en Bolivia y Latinoamérica, donde la confianza y la legitimidad definen la adopción del modelo. PRINCIPIOS DE DISEÑO DEL BACKOFFICE Antes de entrar a las pantallas, es clave fijar los principios que gobiernan todo el diseño: No mostrar jerarquías ni personas debajo del afiliado. Mostrar impacto económico, no estructura de red. Comparar solo contra uno mismo (pasado vs presente). Lenguaje profesional, no motivacional. IA como asistente, no como coach emocional. Con esto, pasamos al diseño pantalla por pantalla. PANTALLA 1 – DASHBOARD / RESUMEN DE IMPACTO Objetivo: Dar al afiliado una visión clara y rápida de su impacto económico y actividad real. Estructura: Header: - Saludo simple (Ej: “Hola, Carlos”) - Periodo activo (Mes actual) Bloque principal (cards grandes): - Impacto económico generado este mes (Bs) - Ingresos estimados / confirmados - Restaurantes activos gracias a tus recomendaciones - Conductores activos recomendados Bloque secundario: - Variación vs mes anterior (+ / – %) - Indicador de estabilidad (actividad constante / irregular) CTA sutil: - “Ver oportunidades” Notas UX: No usar barras de progreso tipo niveles. Todo debe sentirse informativo, no competitivo. PANTALLA 2 – INGRESOS Y ACTIVIDAD Objetivo: Transparencia total sobre cómo se generan los ingresos. Estructura: Tabla principal: - Fecha - Tipo de actividad (restaurante / conductor / usuario) - Descripción (Ej: “Pedidos activos – Restaurante X”) - Monto generado - Estado (pendiente / liquidado) Resumen superior: - Total del mes - Total liquidado - Próxima fecha de pago Filtros: - Por fecha - Por tipo de actividad Notas UX: Nunca mostrar “nivel” ni “equipo”. El foco es trazabilidad. PANTALLA 3 – MI IMPACTO (ANÁLISIS PERSONAL) Objetivo: Mostrar progreso individual sin comparación externa. Estructura: Gráfico principal: - Impacto económico por mes (línea simple) Indicadores: - Crecimiento promedio mensual - Actividad más fuerte (restaurantes / conductores / usuarios) - Días con mayor impacto Mensajes informativos: - “Tu impacto creció 18% respecto al mes pasado” - “Tu mayor aporte viene de restaurantes activos” Notas UX: El afiliado se evalúa a sí mismo, no compite con otros. PANTALLA 4 – OPORTUNIDADES DETECTADAS (IA) Objetivo: Convertir al afiliado en un partner estratégico. Estructura: Lista de oportunidades: - Zona sin cobertura suficiente - Restaurantes activos fuera de la plataforma - Horarios con falta de conductores Cada oportunidad incluye: - Breve explicación - Potencial estimado - Botón “Marcar como atendida” Ejemplo: “Zona Equipetrol – 3 restaurantes con alta demanda aún no afiliados.” Notas UX: La IA sugiere, no ordena. PANTALLA 5 – MISIONES DE VALOR Objetivo: Guiar la acción del afiliado sin presión agresiva. Estructura: Listado de misiones: - Título profesional (Ej: “Expandir cobertura gastronómica”) - Descripción clara - Beneficio asociado (bono, acceso premium, etc.) - Estado (activa / completada) Ejemplos de misiones: - Incorporar 2 restaurantes en una zona nueva - Activar 3 conductores en horarios críticos Notas UX: Evitar contadores tipo “ranking”. Misiones son opcionales . PANTALLA 6 – HISTORIAL Y REPORTES Objetivo: Permitir auditoría personal y confianza. Estructura: Descargas: - Reporte mensual PDF - Historial anual Detalle: - Actividad por periodo - Resumen consolidado Notas UX: Diseño sobrio, tipo reporte financiero. PANTALLA 7 – EDUCACIÓN Y BUENAS PRÁCTICAS Objetivo: Formar afiliados efectivos y profesionales. Estructura: Secciones: - Cómo identificar buenos restaurantes - Cómo explicar el valor de la plataforma - Errores comunes a evitar Formato: - Artículos cortos - Videos breves Notas UX: Educación práctica, no motivacional. El objetivo es maximizar adopción, reducir fricción, evitar errores y proteger la percepción legal y comercial del sistema. RESPUESTA EJECUTIVA DIRECTA El mecanismo de pedir al invitado que escriba manualmente el ID del afiliado en la casilla “¿Quién te invitó?” es funcional y válido, pero no es el método óptimo ni escalable si se utiliza como única forma de atribución. Aunque es correcto desde un punto de vista técnico, presenta fricción, genera errores humanos, reduce la tasa de conversión, incrementa reclamos y puede percibirse como un proceso forzado o poco moderno. Por ello, este método debe existir solo como respaldo, no como canal principal. PRINCIPIO ESTRATÉGICO CLAVE El afiliado nunca debería depender de que otra persona escriba correctamente un dato manual para recibir reconocimiento por su recomendación. Cada paso manual adicional reduce la tasa de registro, aumenta el costo de soporte y deteriora la confianza del usuario. En plataformas modernas y super-apps de escala, la atribución debe ser automática siempre que sea posible. MODELO PROFESIONAL RECOMENDADO: ATRIBUCIÓN MULTICANAL El sistema correcto no se basa en un único método, sino en un modelo de atribución multicanal que combina automatización, respaldo manual y registro asistido. MÉTODO PRINCIPAL RECOMENDADO: LINK O QR DE INVITACIÓN Cada afiliado debe contar con un enlace único y un código QR único de invitación. Ejemplo de enlace: app.com/registro?ref=AF12345 Funcionamiento: - El afiliado comparte su link o QR - El invitado accede al formulario de registro desde ese enlace - El sistema detecta automáticamente el afiliado - El campo “Invitado por” se completa de forma automática - El campo queda bloqueado (no editable) - La atribución queda registrada de forma trazable Ventajas: - Cero fricción - Cero errores humanos - Experiencia moderna - Atribución automática - Fácil defensa legal y técnica Este debe ser el método principal de todo el sistema. MÉTODO SECUNDARIO: CAMPO MANUAL “¿QUIÉN TE INVITÓ?” Este método debe mantenerse como respaldo, pero con reglas claras. Buenas prácticas: - El campo debe ser opcional, nunca obligatorio - Texto sugerido: “¿Alguien te recomendó la app? (opcional)” - Validación automática del ID ingresado - Confirmación visual del afiliado asociado (nombre o alias) - Autocompletado o sugerencias cuando sea posible Casos de uso: - Registro presencial - Registro asistido - Restaurantes con baja digitalización - Invitaciones verbales Este método cubre excepciones sin afectar la experiencia general. MÉTODO AVANZADO: REGISTRO ASISTIDO DESDE BACKOFFICE Para escenarios B2B (restaurantes y conductores), el afiliado debe poder registrar directamente al invitado desde su backoffice. Funcionamiento: - El afiliado inicia el registro del negocio o conductor - El sistema asocia automáticamente la recomendación - No se solicita ID manual al invitado Este método es altamente eficiente y reduce errores, especialmente en procesos comerciales presenciales. REGLA CRÍTICA SOBRE LA REPLICACIÓN DEL MODELO La idea de que “una vez afiliado, la red se replica a todos los negocios activos” es técnicamente válida, pero extremadamente delicada en términos de comunicación. Recomendación estratégica: Nunca comunicar este comportamiento como “replicación de red”. Comunicación correcta: “Tus recomendaciones generan impacto transversal en el ecosistema.” A nivel técnico: - El afiliado no hereda personas - El afiliado hereda impacto económico generado por actividad - La atribución se basa en ventas, uso y retención Esto protege al modelo de estigmas multinivel y riesgos regulatorios. LENGUAJE Y COPY CORRECTO El lenguaje utilizado define la percepción del sistema. Nunca comunicar: - “Para ganar debes poner tu ID” - “Si no pones el ID no ganas” Comunicación correcta: “Si alguien te recomendó la app, puedes registrarlo para que reciba el reconocimiento correspondiente.” Este enfoque refuerza legitimidad y reduce rechazo psicológico . MODELO FINAL RECOMENDADO (RESUMEN EJECUTIVO) Método principal: Link o QR único del afiliado Método secundario: Campo manual opcional Método avanzado: Registro asistido desde backoffice Modelo de atribución: Impacto económico, no personas Estructuras visibles de red: No permitidas CONCLUSIÓN ESTRATÉGICA FINAL El uso exclusivo de IDs manuales es funcional pero frágil. El modelo profesional combina automatización, respaldo manual y atribución basada en actividad real. Este enfoque reduce fricción, aumenta registros, protege la percepción legal y convierte el programa de afiliados en un sistema moderno de partners económicos, alineado con las mejores prácticas de super-apps de gran escala. Este documento aborda una de las preguntas más críticas y sensibles en modelos de negocio con recomendaciones multinivel: cómo justificar transparencia, confianza y legitimidad cuando existen tres niveles descendentes de ganancias, pagos semanales y visualización en tiempo real, pero sin mostrar la composición completa de la red (afiliados/embajadores hacia abajo). El objetivo es entregar una respuesta estratégica, legal, psicológica y de producto para enfrentar estas consultas sin debilitar el modelo ni reforzar el estigma del marketing multinivel. PLANTEAMIENTO DEL PROBLEMA REAL En el modelo planteado existen: - Tres niveles descendentes de recomendación (Nivel 1, Nivel 2, Nivel 3) - Porcentajes diferenciados por nivel - Cierres semanales de pago - Visualización en tiempo real de ganancias De forma natural, algunos embajadores pueden preguntar: - “¿Por qué no puedo ver quiénes están debajo mío?” - “¿Cómo sé que el cálculo es correcto si no veo la red?” - “¿Por qué no tengo derecho a ver mis afiliados hasta el tercer nivel?” Estas preguntas no son malas ni sospechosas: son preguntas legítimas desde una lógica tradicional de redes. El error no está en la pregunta, sino en intentar responderla con el lenguaje clásico del multinivel. PRINCIPIO CLAVE DE RESPUESTA (MUY IMPORTANTE) La transparencia no significa visibilidad total de personas. La transparencia significa trazabilidad, verificabilidad y reglas claras. Este principio debe guiar todo el diseño del sistema y la comunicación. POR QUÉ NO SE DEBE MOSTRAR LA RED DE PERSONAS Existen cuatro razones de peso: Protección de datos y privacidad Los afiliados y negocios son usuarios independientes con derechos de privacidad. Mostrar nombres, contactos o estructuras completas puede violar principios de protección de datos y generar conflictos legales. Evitar competencia interna y conflictos Cuando se muestran redes completas, surgen disputas por “quién pertenece a quién”, captación interna y conflictos entre embajadores. Esto destruye el ecosistema. Eliminación del estigma multinivel La visualización de árboles, niveles y jerarquías refuerza inmediatamente la percepción de MLM clásico, lo que genera rechazo en mercados como Bolivia. Escalabilidad del sistema A gran escala, mostrar redes completas se vuelve inmanejable, pesado y fuente constante de reclamos. CÓMO GARANTIZAR TRANSPARENCIA SIN MOSTRAR PERSONAS La clave está en cambiar el objeto de la transparencia: de personas a actividad. TRANSPARENCIA POR ACTIVIDAD (NO POR PERSONAS) El embajador debe poder ver: - Cantidad de negocios activos por nivel - Cantidad de conductores activos por nivel - Cantidad de usuarios activos por nivel - Actividad generada por nivel (ventas, pedidos, viajes) Ejemplo: “Nivel 1: 4 restaurantes activos – Bs 6.200 en ventas generadas” “Nivel 2: 11 restaurantes activos – Bs 9.800 en ventas generadas” “Nivel 3: 27 restaurantes activos – Bs 14.300 en ventas generadas” Esto permite verificar impacto sin exponer identidades. DESGLOSE MATEMÁTICO AUDITABLE Cada pago semanal debe venir acompañado de un desglose claro: Periodo de cálculo Actividad total por nivel Porcentaje aplicado por nivel Resultado monetario Ejemplo: “Semana 12 Nivel 1: Bs 6.200 x 5% = Bs 310 Nivel 2: Bs 9.800 x 3% = Bs 294 Nivel 3: Bs 14.300 x 2% = Bs 286 Total semanal: Bs 890” Esto elimina dudas sobre el cálculo. HISTORIAL DETALLADO Y EXPORTABLE El embajador debe poder: - Ver histórico semanal - Descargar reportes PDF/Excel - Comparar semanas Esto genera confianza objetiva, no emocional. MENSAJE CLAVE PARA RESPONDER A LAS CONSULTAS Respuesta institucional recomendada: “No mostramos personas porque nuestro modelo no paga por reclutar personas, sino por la actividad económica real que se genera en el ecosistema. Por eso mostramos impacto, no identidades.” Este mensaje debe repetirse en soporte, onboarding y documentos. QUÉ DERECHOS TIENE REALMENTE EL EMBAJADOR Derechos que SÍ tiene: - Ver cómo se calcula cada centavo - Ver actividad generada por nivel - Ver porcentajes y reglas - Ver ingresos en tiempo real Derechos que NO tiene: - Acceso a datos personales de terceros - Control o propiedad sobre otros embajadores - Visualización de estructuras jerárquicas Esto debe quedar claro desde el inicio. OPCIÓN AVANZADA: IDENTIDAD ANONIMIZADA (SI SE QUIERE) Si se desea un punto intermedio, se puede mostrar: - Identificadores anónimos (Ej: Negocio A12, Conductor B07) - Sin nombres, sin contacto Esto refuerza percepción de existencia sin violar privacidad. EDUCACIÓN DESDE EL ONBOARDING Estas reglas no deben descubrirse después. Deben explicarse: - En el onboarding del afiliado - En el backoffice (sección “Cómo se calculan tus ingresos”) - En FAQs claras La sorpresa es enemiga de la confianza. CONCLUSIÓN ESTRATÉGICA FINAL En modelos multinivel modernos, la transparencia no se logra mostrando árboles de personas, sino mostrando matemáticas claras, actividad verificable y reglas inmutables. Al cambiar el foco de “quién está debajo mío” a “qué impacto económico se está generando”, se protege la privacidad, se evita el estigma MLM y se construye un sistema escalable y defendible. Este enfoque permite responder con solidez a cualquier consulta sin debilitar el modelo ni generar fricción innecesaria.
